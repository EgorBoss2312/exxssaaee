# Портал корпоративной базы знаний (прототип ВКР)

Веб-приложение: **чат с ИИ (RAG)**, **база знаний**, **авторизация**, **роли и ограничение доступа к документам**.

**Docker не обязателен.** По умолчанию используется **SQLite** (файл `backend/data/app.db`), эмбеддинги хранятся в JSON, поиск — в Python/NumPy.

---

## Запуск без Docker (рекомендуется)

Нужны **Python 3.9+** и установленные зависимости (первый запуск подкачает модель эмбеддингов — может занять несколько минут).

```bash
cd edda-portal/backend
chmod +x run.sh
./run.sh
```

Откройте в браузере: **http://127.0.0.1:8000** (или http://localhost:8000)

Если порт «не открывается», убедитесь, что в терминале видно строку вроде `Uvicorn running on http://127.0.0.1:8000`, и подождите окончания первого старта (загрузка модели `sentence-transformers`).

**Демо-вход после первого запуска:**

| Email | Пароль | Роль |
|-------|--------|------|
| admin@edda.local | Admin123! | Администратор |
| it@edda.local | It123456! | ИТ (загрузка документов) |
| prod@edda.local | Prod123456! | Производство |
| hr@edda.local | Hr123456! | Кадры |

Пароль администратора при первом создании БД: переменная `SEED_ADMIN_PASSWORD` (иначе используется `Admin123!`).

**Не пускает в систему после обновления кода:** остановите сервер, удалите файл `backend/data/app.db` и запустите снова — база и демо-пользователи создадутся заново.

Демо-логины вида `*@edda.local` раньше отвергались проверкой `EmailStr` (домен `.local` считается служебным). Сейчас вход принимает такие адреса; перезапустите backend после обновления файлов.

## Только интерфейс разработки (Vite на порту 5173)

```bash
cd edda-portal/frontend
npm install
npm run dev
```

Сайт: http://localhost:5173 — запросы к API проксируются на тот же backend (порт 8000 должен быть запущен).

## Запуск с Docker (если установлен)

```bash
cd edda-portal
docker compose up --build
```

Откройте **http://localhost:8000**. В `docker-compose` по умолчанию используется PostgreSQL; для обычной работы приложению достаточно колонки JSON с векторами (расширение pgvector не требуется).

Переменная **`DATABASE_URL`**: для Postgres задайте `postgresql+psycopg://...`; если не задана — в коде по умолчанию SQLite (см. `backend/app/config.py`).

## Ответы ИИ (подключение LLM)

Поиск по базе (**RAG**) работает всегда: из документов, доступных вашей роли, выбираются фрагменты. **Связный ответ** даёт языковая модель, если она настроена; иначе в чате показываются **только вставки фрагментов** (режим «без LLM»).

1. Создайте файл **`edda-portal/backend/.env`** (образец: **`backend/.env.example`**).
2. Выберите один из вариантов (если задано несколько ключей, приоритет: **Gemini → OpenAI → Ollama**).

**Google Gemini** — ключ в [Google AI Studio](https://aistudio.google.com/apikey), в `.env`:

```env
GEMINI_API_KEY=...
GEMINI_MODEL=gemini-2.5-flash
```

Если для выбранной модели закончилась квота (HTTP 429) или имя устарело (404), backend автоматически пробует запасные варианты (`gemini-2.5-flash-lite`, `gemini-flash-latest`). При необходимости задайте другое имя вручную по [списку моделей](https://ai.google.dev/gemini-api/docs/models).

**OpenAI (или совместимый провайдер)** — в `.env`:

```env
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-4o-mini
```

**Локальная Ollama** — установите [Ollama](https://ollama.com), в терминале:

```bash
ollama pull llama3.2
ollama serve
```

В `.env` при необходимости укажите модель и адрес (по умолчанию `http://127.0.0.1:11434` и `llama3.2`):

```env
OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=llama3.2
```

Если backend в **Docker**, а Ollama на компьютере хоста (macOS/Windows), для подключения к Ollama с контейнера задайте, например, `OLLAMA_BASE_URL=http://host.docker.internal:11434` (в Linux может понадобиться другой адрес хоста).

Файл **`backend/.env`** подхватывается **всегда по абсолютному пути** (не зависит от того, из какой папки вы запустили `uvicorn`). После правок **сохраните файл на диск** (Cmd+S / Ctrl+S): если на вкладке редактора есть **точка** у имени файла — изменения ещё не записаны, backend их не видит. Настройки перечитываются при запросах; если режим LLM не обновился — **перезапустите** backend.

Для **Gemini** ключ из Google AI Studio всегда начинается с **`AIzaSy`** (латинская «A» в начале). Если скопировалось без первой буквы, API не сработает.

На странице чата отображается, какой режим активен (Gemini / OpenAI / Ollama / только фрагменты). Ошибки API смотрите в логе консоли сервера.

## Деплой

Сборка фронта в статику и один процесс Uvicorn, либо образ из `Dockerfile`. В продакшене задайте сильный **`JWT_SECRET`**, HTTPS и резервное копирование БД.

**Если на сервере снова «Failed to fetch» при входе (а локально всё ок):**

1. **Один домен и nginx** — запросы к **`/api/`** должны уходить на Uvicorn, а не в статику. Пример: `deploy/nginx-example.conf`. В логах при старте API видно строку `CORS: N origin(s)` — проверьте, что `N` не 0 и в списке есть ваш `https://…`.
2. **Фронт и API на разных URL** — при сборке: `VITE_API_BASE_URL=https://api.ваш-домен.ru` или в `index.html` мета-тег `<meta name="edda-api-base" content="https://api...." />` / `window.__EDDA_API_BASE__`.
3. **Подкаталог сайта** (`https://host.ru/app/`) — в `frontend/vite.config.ts` задайте `base: '/app/'`, пересоберите фронт; либо укажите полный URL API через `VITE_API_BASE_URL`.
4. **CORS** — `CORS_ORIGINS=https://точный-url-фронта` (схема и домен как в адресной строке). Превью: `CORS_ORIGIN_REGEX` в `backend/.env.example`.
5. **HTTPS** — страница по `https://` не может вызывать API по `http://`.
